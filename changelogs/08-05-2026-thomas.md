## Fixed
- API key validation now correctly passes through all network-level errors, not just `NetworkError`.
  - Root cause: `httpx.NetworkError` is a subclass of `httpx.RequestError`, but other subclasses (e.g. `ConnectError`, `RemoteProtocolError`) were not caught, causing valid connectivity failures to surface as rejected keys.
  - Fix: widened the exception catch from `httpx.NetworkError` to `httpx.RequestError` in both the generic and Gemini validation paths in `config.py`.
- `httpx` added as an explicit runtime dependency in `pyproject.toml` (was an implicit transitive dep; now declared directly since `config.py` imports it).
- Process event loop is now properly closed on exit, preventing `ResourceWarning` noise at shutdown.
  - A `_close_loop()` function registered via `atexit` in `runner.py` closes the process-lifetime event loop created by `_get_loop()`. `asyncio.set_event_loop(None)` is called before `loop.close()` so that any finalizer that calls `asyncio.get_event_loop()` during teardown does not receive a closed loop.
- Status animation errors no longer mask real stream exceptions.
  - Root cause: if `_animate_status` raised (e.g. because the event loop was shutting down), `await status_task` in the `finally` block of `_invoke_agent_stream` would propagate that error and swallow the original stream exception.
  - Fix: wrapped `await status_task` in a `try/except` that logs the animation error via `traceback.print_exc()` instead of suppressing it silently, ensuring the original exception propagates correctly.
- Known API errors now show clean, human-readable messages instead of raw tracebacks and JSON blobs.
  - Root cause: all exceptions in `run_agent()` were treated the same — the full traceback was printed to the console and the raw exception string (including verbose provider JSON) was surfaced in the fallback panel.
  - Fix: added `_classify_api_error()` in `runner.py` that matches well-known error patterns and returns a concise `AIResponse` directly, bypassing the generic fallback. Three cases are handled: rate limits / quota exhaustion (429 / `RESOURCE_EXHAUSTED`), authentication failures (401 / 403 / `API_KEY_INVALID`), and provider unavailability (503 / `UNAVAILABLE`). Unexpected errors still print the full traceback so they remain debuggable.
